## 1. 対象者・システムの目的

### 1.1. 対象者

QINeS-GAIのフロントエンドに関する開発メンバ(保守運用者も含む)

### 1.2. システムの目的

ユーザとAIエージェントが対話を繰り返すことで仕様書やコード・レポートなどの成果物を生成するためのシステムである。ユーザはAUTOSARをはじめとした車載システムに関する膨大なドキュメントを基に対話ができる。ユーザはMarkdownやPowerPoint・PDFなどの形式で成果物をダウンロードしていくことがゴールとなる。

## 2. 使用している言語・フレームワーク

- Next.js (App Router)
- TypeScript
- TailwindCSS

## 3. 技術的な課題とソリューション

### 3.1. 基本方針

以下を満たすライブラリを採用する

- ライセンス：MITやLGPLなどプロダクト配布/納品先への利用物コード公開が不要であること。
- 最終更新：直近1年以内に更新されていること。

### 課題1. markdownをプレビューしチャットや成果物を表示する

#### 1.1. 採用したソリューション

[react-markdown-preview](https://github.com/uiwjs/react-markdown-preview)

- 理由：独自 CSS やコンポーネントを利用し、出力する要素ごとにスタイルをカスタマイズでき、拡張性に富んでいるため。

#### 1.2. 他に検討したソリューション

[react-markdown](https://github.com/remarkjs/react-markdown)

- react-markdown-preview がこれをラップしているため機能が少なくなるため不採用。

### 課題2. バックエンドサービスとのAPI通信

#### 2.1. 採用したソリューション

[openapi-zod-client](https://github.com/astahmer/openapi-zod-client)

- 理由：コマンド一つで OpenAPI を基に API クライアントを生成してくれる。また、Zod を利用したインタフェースを生成するため型安全なシステムを開発可能。バックエンドと並行開発をする都合上、API クライアントをシステマチックかつ迅速に生成できるため採用した。

#### 2.2. 他に検討したソリューション

[openapi-generator-cli](https://openapi-generator.tech/docs/installation/)

- 生成されるコードが型安全ではないため不採用。

### 課題3. バックエンドサービスのモック化

#### 3.1. 採用したソリューション

[Prism](https://stoplight.io/open-source/prism)

- 理由：完成していない API を実行可能とすることでフロントエンド開発をスムーズに実施するために採用。
- デメリット：ファイルアップロードやストリーム処理に関するモック化がサポートされていない。

#### 3.2. 他に検討したソリューション

なし

### 課題4. 状態管理

#### 4.1. 採用したソリューション

[jotai](https://jotai.org/)

- 理由：本システムでは複雑な状態管理が不要であるため、より軽量な本ライブラリを採用。

#### 4.2. 他に検討したソリューション

[Recoil](https://github.com/facebookexperimental/Recoil)

- 更新が止まったため不採用。

[Redux](https://redux.js.org/)

- 学習コストが高いことで知られている。本件では会社をまたぐ引き継ぎが発生するため不採用。

### 課題5. 巨大なPDFをレンダリングしたい

#### 5.1. 採用したソリューション

[react-pdf](https://github.com/wojtekmaj/react-pdf)

- 理由：PDFレンダリングが簡易に行えることと、後述の仮想レンダリングとの相性が良いため。

[react-virtuoso](https://virtuoso.dev/)

- 理由：扱うPDFが巨大(最大約2000ページ)であるため、一括でレンダリングをするとブラウザがクラッシュしてしまう問題があった。それを回避するために仮想レンダリングツールを採用。

#### 5.2. 他に検討したソリューション

Embed Google Docs PDF viewer(iframe)

- Googleが提供するPDF viewerを埋め込むことが可能。軽量かつ動作も安定しているので採用候補に挙がった。しかし、IP制限をかけたS3ファイルへのアクセスができないため不採用。また、選択して部分翻訳やチャットに送信するなどの機能を将来的に追加する可能性がある。そのような仕組みには対応していないことも不採用の要因である。

[react-window](https://github.com/bvaughn/react-window)

- react-virtuoso同様に仮想レンダリングをサポートする。PDFを拡大・縮小する際にレイアウトが崩れる問題が発生したため不採用。

### 課題6. レポートの内容をPDFとしてダウンロードしたい

#### 6.1. 採用したソリューション

[puppeteer](https://pptr.dev/)

- 理由：サーバサイドでPDFを生成できるツールでありクライアントへの負荷を小さくできるため採用。
- 懸念：ヘッドレスブラウザでHTMLを描画してそれをPDFに変換している。開発/本番環境のOSではchromiumをサポートしていないためfirefoxを利用している。

[Markdown-it](https://github.com/markdown-it/markdown-it)

- 理由：Markdownの文字列からHTML要素を生成するために利用する。生成したHTMLに任意のスクリプトやCSSを付与できるなどの拡張性の高さから採用。

#### 6.2. 他に検討したソリューション

[react-pdf](https://github.com/wojtekmaj/react-pdf)

- PDFレンダリングをするために利用しているため採用したかった。しかし、MarkdownやHTMLをPDFにする際には一度canvas化してからPDFにする必要がある。そのため、全体が画像のように扱われてしまうことが欠点。また、canvas化する都合上、画面右半分に成果物がある場合は、生成されるPDFの横幅が画面半分までになってしまう。これらが解決できなかったため不採用。

## 4. テスト自動化の方針

本開発では、フロントエンドに範囲を絞った単体テストのみ自動化する。フロントエンドとバックエンドを通じて行う結合テストは手動での実行とする。また、開発効率向上のため以下の条件を満たす場合はテストコードを作成しなくても良いものとする。ただ、テストケースの存在を検知するために`test.todo`として残しておくこと。

- ブラウザ上での動作確認が簡単であること。
- 不具合が発生した場合の影響が小さいこと。
- 不具合が発生していることを検知しやすいこと。
- テストコード作成に時間がかかるもの。

## 5. CI/CD

### 5.1. CI

リポジトリへのコミット時とプッシュ時にそれぞれCIを行っている。

- コミット時
  - pre-commitを利用する。
  - 以下を実行する。
    - prettierとeslintを用いたフォーマットと静的解析。
- プッシュ時
  - pre-pushを利用する。
  - 以下を実行する
    - OpenAPI Clientの生成。
    - prettierとeslintを用いたフォーマットと静的解析。
    - 自動テストの実行
    - ライセンスチェック
    - ビルド

### 5.2. CD

バックエンドを含めた全体を考慮したCDを提供する必要があるため、フロントエンド単体ではCDを採用していない。

## 6. エラーハンドリングの方針

2025年3月時点では正常系の機能重視で開発を行っている。また、バックエンドとフロントエンド間でどのようなメッセージを表示すべきかを議論しきれていない。今後メッセージのあるべき姿を定義し追加で改修を行う必要がある。`hooks/useError.ts`にメッセージテンプレートを用意しているため、それをメンテナンスしていくことが望ましい。
